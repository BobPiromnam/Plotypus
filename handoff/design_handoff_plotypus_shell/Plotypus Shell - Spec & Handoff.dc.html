01 — Architecture audit

Five control surfaces, no clear owner

Historical additions left settings, navigation, commands and status scattered across competing regions. The same kind of control appears in more than one place, and the canvas — the product’s entire purpose — is a non-default tab.

Today’s surfaceWhat it holdsProblem
Top ribbon .app-ribbonProject, Data, Export, Appearance (map style + language)Mixes global commands with render settings; 86px tall.
Left sidebar .sidebar (376px)Only “Legend & markers” categories + hidden layout inputsNear-empty permanent column; eats table width.
Table tabs .table-tabsProject points · Map regions · Translate · Map · QualityMap (the destination) is one tab among five.
Right panel .properties-panelDisplay, Interaction, SelectionDuplicates document defaults that also live elsewhere.
Summary chips #workspaceSummaryMetricsRows · mapped · callouts · issuesPassive counts and actionable problems look identical.
02 — Alternatives considered

Three structures, one chosen

A · Left icon rail

VS Code-style vertical rail as primary nav.

Rejected: reintroduces a permanent left column the brief asks to remove; steals table width.
B · Mode toggle + modal panels

Edit / Preview modes, settings in dialogs.

Rejected: hides dense inline editing behind modals; breaks keyboard flow.
C · Horizontal workspace nav ✓

One stable top nav; canvas default; no left column.

Chosen: removes the sidebar, keeps tables full-width, makes Map the home destination.
03 — Recommended architecture

One persistent shell, five owners

Three stacked horizontal bands above one body. Each band owns exactly one job; the body is the active workspace plus a contextual Properties panel. Categories & markers becomes its own workspace, vacating the left sidebar entirely.

1 · Command barbrand · Project · Data · Map style · Map language · Export
2 · Workspace navMap · Project points · Categories · Regions · Translate · Quality
3 · metrics4 · review
active workspace body
5 · Properties
Global commands live only in the command bar — they never change per workspace.
Tab-specific tools live in a local toolbar at the top of each workspace body (or floating on the canvas).
Passive metrics (rows/mapped/regions) sit quietly on the right of the nav band.
Actionable problems collect in one ochre “N to review” cluster that routes to fixes.
Selection properties are the only thing in the right panel — contextual to the active workspace.
Map stays the default landing workspace and the largest surface.

07 — Component specifications

Build details

Command bar
56px, --panel bg, sticky, 1px bottom border. Left: brand + grouped icon buttons (Project / Data) with a mono group caption under each, 1px dividers between groups. Right: Map style select, Map language segmented (EN/FR/Bi), accent Export ▾. Buttons 12.5px, icon 15px stroke 1.8.
Workspace nav
role="tablist" on --surface band. Six items, 16px icon + 13px label, active = accent fill/white/800. Map first & default.
Status spine
Right of nav band. Passive metric = mono value + muted word, no border. Actionable = one ochre pill N to review → routing to Map quality with the relevant filter.
Project table
Column bands (Identity · Classification · Coordinates · Output) via a grouping header row. Name column sticky-left. Coordinates mono & right-aligned. Validation badge only on problem rows. Row 40px, zebra #faf8f3.
Canvas workspace
--workspace-canvas-bg #ebe7dc pasteboard, centered white print-frame with crop marks, hatched placeholder. Floating toolbar (zoom / Fit / Fit map + labels / Labels only) top-center; docked quality readout bottom-center with Locate next.
Properties panel
Header = type glyph + what’s selected. Sections shown only if relevant to the active workspace: Map → Display + Interaction; Project points → selected row fields; Categories → marker shape/colour; Regions → colour actions; Translate → keyboard help.

13 — Implementation order

File-oriented sequence

STRUCTURE · index.html
  1. Split header into .command-bar + .workspace-nav.
  2. Move table tabs into the nav; promote Map to first/default.
  3. Lift #categoryList out of .sidebar into a Categories workspace; delete the sidebar.
  4. Wrap each pane in [data-workspace]; tag Properties sections [data-props-context].
STYLING · style.css
  1. Add new tokens to :root.
  2. Command bar + nav band layout; remove 3-col .app-shell grid’s left track.
  3. Status spine: passive vs [data-summary-state] styling.
  4. Mono + right-align numeric cells; zebra; sticky name; danger separation; focus rings (already present).
  5. Responsive: 1280 / 1024 rules per §5.
BEHAVIOUR · app.js
  1. Point nav clicks at existing setActiveDataTab(); default to Map on load.
  2. Render Properties section visibility from active workspace (data-props-context).
  3. Split summary into passive metrics + #workspaceReviewBtn in renderWorkspaceSummary().
  4. Wire Locate buttons from updateStatus() / renderQualityMetrics().
  5. Guard: nav/filter/select stay lightweight; never call the plot path.
RISKY · verify carefully
  1. Removing the sidebar grid track — check no JS reads .sidebar layout.
  2. Map language mirroring across the two selects.
  3. Don’t touch #mapSvg render or export paths.
  4. Run the smoke + label-geometry tests after.

Acceptance criteria

✓ Nav stays stationary between workspaces.
✓ No permanent left sidebar.
✓ Categories & markers fully accessible (own workspace).
✓ Map is the primary, default workspace.
✓ Properties never show irrelevant controls.
✓ Passive metrics vs actionable issues clearly distinct.
✓ Tables keep adequate width (≥760px) & never shrink.
✓ No control overlap at 1440 / 1280 / 1024.
✓ Keyboard focus always visible.
✓ Map render & export output unchanged.
✓ Lightweight actions never trigger a re-plot.
✓ App remains usable fully offline.
